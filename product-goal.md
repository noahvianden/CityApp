# Cityprint Product Goal

Cityprint is a mobile app for personal city exploration. The app reveals a city by movement, hides places until the area around them has been discovered, and lets the user save memories that stay private by default.

## Product Summary

Cityprint is not a navigation app. It is not a review platform, tourist guide, fitness tracker, or social route-sharing app. Its job is to turn walking into discovery and keep the resulting memories and location data under the user's control.

The product goals are:

- select a city
- view a partially hidden city map
- reveal map areas by real-world movement
- show places only in revealed areas
- save personal memories
- view city and district progress
- create privacy-preserving recaps
- control location and memory data

## Core Loop

1. The user selects a city or continues a previously started one.
2. The app shows a mostly hidden map with fog-of-war.
3. The user walks in the real world.
4. Movement reveals nearby map areas.
5. Places become visible only when their area is revealed.
6. The user opens a place, saves it, or adds a personal memory.
7. Progress, district discovery, and recap views summarize what was found.
8. Privacy controls keep exact routes and sensitive data hidden unless the user explicitly allows more.

## Core Entities

- `city`: an explorable city containing map areas, districts, places, memories, and progress data.
- `map_area`: a spatial segment of the map with hidden, partially revealed, or revealed state.
- `fog_of_war`: the visual and interaction state for unexplored map areas.
- `revealed_area`: a map area unlocked by movement.
- `route`: the path generated by user movement and used to reveal areas.
- `place`: a discoverable location such as a cafe, restaurant, park, gallery, shop, bar, market, viewpoint, quiet spot, or culture venue.
- `memory`: a user-created personal entry linked to a place, route, or map area.
- `district`: a contiguous city area used for orientation and progress.
- `recap`: a privacy-preserving summary of exploration activity.

## Map System

The map must be readable as an urban map, not as a visible tile grid. It should use layered city structure:

- base map
- geography
- districts
- streets
- blocks
- discovery layer
- fog of war
- labels
- places
- memories
- interaction layer

Requirements:

- streets must form a plausible network
- blocks must follow street geometry
- parks, water, places, and districts must support orientation
- districts must be contiguous
- places must only appear in revealed areas
- memories must be visually distinct from places

## Fog of War

Fog states:

- fully hidden
- partially revealed
- recently revealed
- fully revealed

Requirements:

- hide unexplored areas
- prevent interaction with places under fog
- reduce fog when an area is revealed
- keep the transition between hidden and revealed clear
- avoid a visible tile grid as the primary visual

## Route Reveal

Requirements:

- show the walk route
- show the current location
- reveal the area around the route
- include adjacent streets and blocks when relevant
- trigger new place visibility when a place enters a revealed area
- allow memory creation during or after a walk

## Privacy Requirements

- exact routes are hidden by default in recaps
- sensitive places can be hidden
- home and work areas can be blurred
- memories can be exported if the feature is enabled
- backup remains user-controlled
- location and memory data should stay local first
- precise location should only be used when needed for active walks

## Acceptance Criteria

- users understand that movement reveals the city
- users understand that places only appear in revealed areas
- users can distinguish places from memories
- users can start and pause a walk
- users can save a memory
- users can open place details
- users can view progress
- users can find privacy controls
- the map appears plausibly urban
- the app does not behave like a navigation, review, or tourist-guide app

## Current Technical Priority

The main technical line is the location pipeline:

- simulated samples
- GPS samples
- permissions
- background behavior
- reveal logic driven by those inputs
- privacy-safe handling of what is stored and shown

That is the next area that must stay coherent while the product grows.
